# 프로젝트 복구 계획
# Project Recovery Plan

## 🚨 현재 상황 분석

### 발견된 문제점
1. ✅ DMZ 로고 파일은 존재함 (`assets/images/logos/dmz-logo.png`)
2. ✅ 모든 데이터 파일은 새 위치에 존재함 (`assets/data/core/`, `assets/data/conflicts/`)
3. ✅ **원본 파일들이 모두 `web-layout/` 폴더에 그대로 남아있음**
4. ❌ 새로 복사된 파일들의 경로가 잘못 업데이트됨
5. ❌ 시각화 차트 작동 안 함
6. ❌ Country selector 작동 안 함

### 근본 원인
- 파일을 **이동(mv)**하지 않고 **복사(cp)**했기 때문에 원본과 새 버전이 공존
- 새 버전의 경로는 업데이트되었지만 상대 경로 깊이가 맞지 않음
- 원본 파일들이 여전히 작동 가능한 상태로 남아있음

## ✅ 복구 전략

### 전략 1: 원본 복원 (권장) ⭐
**장점**:
- 빠른 복구
- 검증된 작동 상태
- 위험도 낮음

**단점**:
- 최적화된 구조 포기
- 파일 구조 개선 없음

### 전략 2: 새 구조 수정 (현재 진행)
**장점**:
- 최적화된 구조 유지
- 장기적 이점

**단점**:
- 복잡한 디버깅 필요
- 시간 소요

## 🎯 권장 접근 방식

**하이브리드 방식**: 원본 유지 + 점진적 개선

1. **즉시 복구**:
   - 원본 `web-layout/` 폴더 사용
   - 새로 이동된 데이터 파일 활용
   - 원본 HTML/JS/CSS는 그대로 두고 경로만 수정

2. **점진적 개선**:
   - 원본이 작동하는 상태에서 하나씩 개선
   - 각 페이지를 테스트하며 이동

## 📋 복구 실행 계획

### Phase 1: 긴급 복구 (15분)
1. ✅ 원본 파일 확인 완료
2. 원본 HTML/JS/CSS 파일의 데이터 경로만 수정
3. 기본 기능 복구 확인

### Phase 2: 경로 최적화 (30분)
1. `web-layout/` 내부에서 데이터 경로 수정
2. `data/` → `../assets/data/` 로 변경
3. 이미지 경로 수정

### Phase 3: 테스트 (15분)
1. 메인 페이지 테스트
2. 시각화 차트 테스트
3. Country selector 테스트
4. 모든 데이터 페이지 테스트

### Phase 4: 정리 (선택사항)
1. 작동 확인 후 `src/` 폴더 삭제 또는 보관
2. 문서 업데이트

## 🔧 즉시 실행할 작업

1. **원본 파일 기반 복구**
   - `web-layout/` 폴더의 HTML 파일 경로 수정
   - 데이터 참조: `data/` → `../assets/data/core/`
   - 이미지 참조: `data/logo.png` → `../assets/images/logos/dmz-logo.png`

2. **JavaScript 데이터 로딩 수정**
   - `web-layout/js/` 파일들의 fetch 경로 수정
   - 절대 경로를 상대 경로로 변경

3. **검증**
   - 브라우저에서 `web-layout/index.html` 열기
   - 모든 기능 작동 확인
